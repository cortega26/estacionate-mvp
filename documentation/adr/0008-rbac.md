# ADR 0008: RBAC — Roles, Permisos y Enforcement

**Estado:** Aceptado
**Fecha:** 2026-04-27

**Nota Fase 1:** Los permisos financieros aquí descritos son para
demo/simulador o fases futuras bloqueadas. La fase habilitada no incluye pagos
integrados, payouts ni custodia de fondos de comunidades.

---

## Contexto

El schema actual define un enum `Role` con valores: `admin`, `support`, `building_admin`, `concierge`, `sales_rep`. El frontend usa adicionalmente `resident`. No existe una matriz formal de permisos por entidad ni un helper de enforcement centralizado. Los endpoints verifican `req.user.role` en línea con lógica repetida e inconsistente.

Síntomas actuales:

- `TECH_SPEC.md` documenta roles ADMIN/RESIDENT/GUARD — difiere del schema real.
- No hay tests negativos que verifiquen que un `concierge` no puede acceder a `/api/admin/users`.
- Los roles no están scoped por edificio (ver ADR 0007).

---

## Decisión

### Roles canónicos

| Rol                | Alcance                    | Descripción                                                                      |
| ------------------ | -------------------------- | -------------------------------------------------------------------------------- |
| `super_admin`      | Global                     | Acceso total; break-glass auditado                                               |
| `platform_ops`     | Global operacional         | Gestión de edificios, incidentes, sin payouts productivos                        |
| `platform_support` | Global / ticket            | Acceso mínimo necesario; just-in-time                                            |
| `finance`          | Global financiero          | Demo/futuro bloqueado: payments, payouts, refunds; dual approval en montos altos |
| `sales_manager`    | Global comercial           | Sales reps, leads, sin PII de residentes                                         |
| `sales_rep`        | Sus edificios asignados    | Comisiones propias, sin reservas personales                                      |
| `legal_viewer`     | Global read-only           | Contratos, T&C, audit; acceso justificado                                        |
| `building_admin`   | Edificio(s) por membership | Operación del edificio, reglas, precios, usuarios del edificio                   |
| `committee_viewer` | Edificio — read-only       | Dashboard y reportes; sin controles destructivos                                 |
| `concierge`        | Edificio por membership    | Verificación de acceso, check-in/out, incidentes                                 |

Los `Resident`s tienen su propio actor type y no usan el enum `Role` de `User`. Ver ADR 0009.

| Entidad        | super_admin | building_admin (scoped) | concierge (scoped) | finance | platform_support |
| -------------- | ----------- | ----------------------- | ------------------ | ------- | ---------------- |
| Building       | CRUD        | RU (propio)             | R                  | R       | R                |
| User (interno) | CRUD        | R (propio edificio)     | —                  | —       | R                |
| Booking        | CRUD        | CRUD (scoped)           | R + transition     | R       | R                |
| Payment        | CRUD        | R                       | —                  | RU      | R                |
| Payout         | CRUD        | R                       | —                  | CRUD    | —                |
| AccessEvent    | CRUD        | R                       | C                  | —       | R                |
| AuditLog       | R           | R (scoped)              | —                  | R       | R                |

### Middleware de enforcement: `requireRole()`

Implementar en `backend/src/middleware/auth.ts` (o archivo separado):

```typescript
// Pseudocódigo — la implementación real va en backend/src/middleware/
function requireRole(allowedRoles: Role[], options?: { buildingScoped?: boolean }) {
  return (req, res, next) => {
    const actor = req.user // del JWT
    if (!actor || !allowedRoles.includes(actor.role)) return res.status(403).json(...)
    if (options?.buildingScoped) {
      // derivar buildingId del membership del actor, no del request
      const buildingId = resolveBuildingScope(actor)
      if (!buildingId) return res.status(403).json(...)
      req.buildingScope = buildingId
    }
    next()
  }
}
```

Todo endpoint debe declarar su `requireRole()` explícitamente. No hay rutas sin verificación de rol.

### `resolveBuildingScope(actor)`

Helper que:

1. Si el actor tiene rol global (`super_admin`, `platform_ops`, `finance`, etc.) → devuelve `null` (sin filtro, scope global).
2. Si el actor tiene rol scoped (`building_admin`, `concierge`, `sales_rep`) → busca `UserBuildingMembership` en Redis/DB y devuelve el `buildingId`.
3. Si el actor es `Resident` → devuelve `resident.unit.buildingId` desde el JWT.
4. Si no encuentra scope → devuelve error 403.

El `buildingId` derivado se inyecta en `req.buildingScope` y es lo que usan las queries — nunca `req.params.buildingId` ni `req.body.buildingId` sin validación contra `req.buildingScope`.

---

## Alternativas descartadas

| Alternativa                                    | Por qué se descartó                                                  |
| ---------------------------------------------- | -------------------------------------------------------------------- |
| Verificación de rol inline en cada handler     | Ya está haciendo esto; es inconsistente y no testeable como unidad   |
| Permisos hardcodeados por endpoint sin helper  | No escala; cambia de rol requiere buscar todos los endpoints         |
| ABAC completo (Attribute-Based Access Control) | Over-engineering para el tamaño actual; RBAC scoped es suficiente    |
| ACL por recurso                                | Requiere tabla de ACL; RBAC por membership es más simple y auditable |

---

## Criterios de aceptación (gate para implementación)

- [ ] `requireRole()` middleware existe en `backend/src/middleware/` y acepta array de roles + flag `buildingScoped`.
- [ ] `resolveBuildingScope()` existe y es usado por todos los endpoints scoped.
- [ ] Todos los endpoints `/api/admin/**` usan `requireRole()` con roles explícitos.
- [ ] Todos los endpoints `/api/concierge/**` usan `requireRole(['concierge', 'building_admin'], { buildingScoped: true })`.
- [ ] Test negativo: `concierge` recibe 403 en `/api/admin/users`.
- [ ] Test negativo: `building_admin` de Building A recibe 403 en recursos de Building B.
- [ ] Test negativo: `sales_rep` sin asignación a un edificio recibe 403 en ese edificio.

---

## Consecuencias

- El enum `Role` en Prisma debe actualizarse para incluir `super_admin`, `platform_ops`, `finance`, `sales_manager`, `legal_viewer`, `committee_viewer`. El valor `admin` existente se mapea a `super_admin` o `building_admin` según el contexto — requiere migración de datos.
- Los endpoints existentes que hacen `if (req.user.role === 'admin')` deben actualizarse al nuevo middleware.
- Los tests actuales de auth que verifican roles deben actualizarse a los nuevos valores del enum.
